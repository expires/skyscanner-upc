# Roam — V5: Power BI Integration
> Real-time travel intent analytics. Backend streams engagement data to Power BI for live business intelligence dashboards.

## What's New in V5

V4: Centralised Express backend + MongoDB Atlas.
V5: Backend forwards engagement events and intent scores to Power BI streaming datasets in real time. Every scroll becomes a data point on a live dashboard.

---

## Why Power BI

Roam generates a continuous stream of behavioural intent signals. Power BI turns that stream into a live business intelligence layer — showing which destinations are trending, which engagement signals drive bookings, and how intent scores distribute across users in real time.

This is the data product underneath the consumer app.

---

## Architecture

```
[Chrome Extension]
  - Engagement events captured locally
        ↓ HTTPS
[Express Backend - Railway]
  - Receives events + scores
  - Stores in MongoDB (V4)
  - Forwards to Power BI streaming endpoints (V5 addition)
        ↓ POST
[Power BI Streaming Datasets]
  - events dataset — raw engagement stream
  - scores dataset — intent score updates
        ↓
[Power BI Dashboard]
  - Live tiles updating in real time
  - Destination rankings
  - Engagement signal breakdown
  - Intent score distribution
```

## Backend — V5 Changes

### New file: src/services/powerbi.ts

```typescript
const EVENTS_URL = process.env.POWER_BI_EVENTS_URL!;
const SCORES_URL = process.env.POWER_BI_SCORES_URL!;

export async function streamEventsToPowerBI(
  events: EventDocument[]
): Promise<void> {
  if (!EVENTS_URL) return; // graceful skip if not configured

  const rows = events.map(e => ({
    destination: e.destination,
    country: e.countryCode,
    countryCode: e.countryCode,
    eventType: e.eventType,
    platform: e.platform,
    dwellSeconds: e.duration ? Math.round(e.duration / 1000) : 0,
    timestamp: new Date(e.timestamp).toISOString(),
    deviceId: e.deviceId,
  }));

  try {
    await fetch(EVENTS_URL, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(rows),
    });
  } catch (err) {
    console.warn('[Roam] Power BI events stream failed silently', err);
    // never block main flow on analytics
  }
}

export async function streamScoresToPowerBI(
  deviceId: string,
  scores: ScoreDocument[]
): Promise<void> {
  if (!SCORES_URL) return;

  const rows = scores.map(s => ({
    destination: s.destination,
    country: s.country,
    countryCode: s.countryCode,
    intentScore: s.score,
    rawScore: Math.round(s.rawScore),
    totalDwellSeconds: Math.round(s.breakdown.totalDwell / 1000),
    postCount: s.breakdown.postCount,
    saves: s.breakdown.saves ?? 0,
    rewatches: s.breakdown.rewatches,
    profileClicks: s.breakdown.profileClicks,
    timestamp: new Date().toISOString(),
    deviceId,
  }));

  try {
    await fetch(SCORES_URL, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(rows),
    });
  } catch (err) {
    console.warn('[Roam] Power BI scores stream failed silently', err);
  }
}
```

### Updated routes/events.ts

```typescript
import { streamEventsToPowerBI } from '../services/powerbi';

router.post('/', async (req, res) => {
  const { deviceId, events } = req.body;

  // store in MongoDB (V4)
  await eventsCollection.insertMany(
    events.map(e => ({ ...e, deviceId, createdAt: new Date() }))
  );

  // stream to Power BI (V5) — fire and forget
  streamEventsToPowerBI(events.map(e => ({ ...e, deviceId }))).catch(() => {});

  res.json({ ok: true });
});
```

### Updated routes/scores.ts

```typescript
import { streamScoresToPowerBI } from '../services/powerbi';

router.post('/', async (req, res) => {
  const { deviceId, scores } = req.body;

  // upsert in MongoDB (V4)
  for (const score of scores) {
    await scoresCollection.updateOne(
      { deviceId, destination: score.destination },
      { $set: { ...score, deviceId, lastUpdated: Date.now() } },
      { upsert: true }
    );
  }

  // stream to Power BI (V5) — fire and forget
  streamScoresToPowerBI(deviceId, scores).catch(() => {});

  res.json({ ok: true });
});
```

## Key Design Decisions

**Fire and forget:** Power BI streaming never blocks the main request/response cycle. If the Power BI endpoint is down, the extension still works perfectly. Analytics is additive, never critical path.

**No PII:** deviceId is an anonymous UUID. No names, no emails, no accounts. GDPR-safe by design.

**Graceful degradation:** If `POWER_BI_EVENTS_URL` is not set in `.env`, streaming is silently skipped. The backend works identically with or without Power BI configured.

**Batching:** Events are sent as arrays, not individually. One POST per user session flush, not per event — avoids rate limits on Power BI's streaming API (10,000 rows/hour on free tier).

---

## Rate Limits

Power BI free tier streaming limits:
- 10,000 rows/hour per dataset
- 5 pushes/second per dataset

At typical usage (one event every 2-3 seconds of scrolling), a single user generates ~1,200 events/hour — well within limits. Batching per session flush keeps push frequency low.
